# Distribute — Safely push content across 12+ platforms

When user says `/distribute` or "distribute the new tools" or "post the articles":

## Phase 0: ALWAYS check status FIRST (never skip)

```bash
cd "/Users/mdgolamkibriaemon/Projects/Teamz Lab Projects/teamz-projects/teamzlab-tools"

# What's in the draft queue? What's published?
python3 scripts/distribute/distribute.py drafts      # pending articles
python3 scripts/distribute/distribute.py status      # history
python3 scripts/distribute/distribute.py safety      # rate-limit dashboard

# Video stockpile (CRITICAL — blocks renders at 30 unposted)
node scripts/distribute/remotion/render-batch.js --status
```

Never proceed without reading these four outputs. The rate-limit dashboard + video stockpile dictate what's actually possible tonight.

## Phase 1: If user has new article(s) that need drafting

Article must have frontmatter with `title:`, `slug:`, `tags:`, `canonical_url:`, `og_image:`. The `title:` field is MANDATORY — without it, `draft` rejects with "no title in frontmatter".

```bash
python3 scripts/distribute/distribute.py draft scripts/distribute/articles/NAME.md \
  --priority high --series BATCH_NAME
```

Append, never overwrite. Article files live in `scripts/distribute/articles/` (symlinked to submodule).

## Phase 2: Post to web platforms (rate-limited, auto-safe)

```bash
# One-shot: post next best draft across all safe platforms
python3 scripts/distribute/distribute.py next --count 3

# Or publish already-queued safe platforms
python3 scripts/distribute/distribute.py flush
```

Rate limits per platform (from `config.json` — do not override):
- devto, hashnode, tumblr, youtube: 1/day each
- blogger, wordpress, bluesky, github_discussions, telegraph: 3h cooldown between posts
- mastodon, substack, pinterest: special — see issues below

**Known platform issues (as of 2026-04-14):**
- **Mastodon:** account login disabled — fix at mastodon.social/@teamzlab
- **Substack:** missing lib — `pip install python-substack`
- **Pinterest:** trial API can't post to production — needs tier upgrade

## Phase 3: YouTube Shorts — two paths (don't confuse)

**Path A (default, safe, automatic):**
`distribute.py next` includes `youtube` as a platform → auto-calls Remotion autopilot → renders 1 Short + uploads. Rate-limited 1/day. Already handled by Phase 2.

**Path B (manual — for brand-new niches content-engine can't handle):**
```bash
# 1. Check stockpile (BLOCKS at 30 unposted without --force)
node scripts/distribute/remotion/render-batch.js --status

# 2. If user wrote custom video-plans.json entries
node scripts/distribute/remotion/render-batch.js --from-plans --count N [--force]

# 3. Upload rendered MP4s (respects 1/day YouTube limit)
node scripts/distribute/remotion/youtube-autopilot.js
```

**WARNING — `content-engine.py` limitations:**
- It fuzzy-matches the input keyword against the existing tool search-index
- GOOD for "pick 5 random tools, generate Shorts" workflow
- BAD for "generate a Short for this specific new article" — returns garbage (unrelated tools matched)
- For new articles, **write `video-plans.json` entries manually** using the schema at `render-batch.js:442` — never let content-engine generate for new niches

**NEVER manually overwrite `video-plans.json`** unless you know what you're doing — it's auto-generated by content-engine. To safely add custom entries, write a standalone JSON file and use `--plans-file PATH` if supported, otherwise append to the existing file.

## Phase 4: After posting

```bash
python3 scripts/distribute/distribute.py status | head -20

python3 scripts/distribute/distribute.py safety

```

## Hard gates for autonomous runs

1. **NEVER bypass `--force` checks on renderer without user permission** — the 30-video stockpile limit exists for a reason (YouTube spam signal).
2. **NEVER post more than rate limits allow** — platforms ban for velocity. The config.json caps are not suggestions.
3. **NEVER use `post "title" file.md --platforms all`** — this bypasses rate-limit safety. Always use `draft` + `next`.
4. **Check article `title:` in frontmatter BEFORE running draft command** — otherwise confusing error.
5. **If stockpile ≥ 30 AND user wants new renders, default to posting existing first.** Only force-render on explicit user override ("force", "override", "just do it").

## Typical flow (green-light case)

1. Health check (4 commands in Phase 0)
2. Any new article? Register via `draft` → append to queue
3. Run `next --count N` with N = number of articles to cycle through
4. Report back: what posted, what queued, what blocked, what failed
5. If YouTube Shorts needed but content-engine broken for niche → write manual video-plans.json
6. Exit: tell user tomorrow's rate budget + what will auto-post next run
